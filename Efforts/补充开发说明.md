一、 核心流程与规则的遗漏
1. Ingest 来源类型分类与路由：
◦ 当前的 ingest.py 仅仅区分了 .md 和 .pdf 的文件扩展名。
◦ 遗漏：《补充要求.md》中明确规定了优先级路由机制（检查 type: personal-writing 或路径 raw/personal/ 走「个人写作」流程，检查 type: pdf-reference 走「PDF 参考」流程，其余走外部标准流程）。
2. 缺失 Frontmatter 时的 fallback 规则：
◦ 当前没有处理无 Frontmatter 的逻辑。
◦ 遗漏：要求遇到无 Frontmatter 的文件不中断 INGEST，尝试提取 title、生成留空的 source、使用文件系统时间作为 date，并在 log.md 记录警告。
3. 概念和实体页面的更新闭环（对齐与校验）：
◦ 遗漏：当前我们只是用脚本生成了一个 ## Concepts Extracted的空壳骨架。缺少真正的概念名称对齐校验（检查已有的 wiki/concepts/*.md 及其 aliases），也没有根据提取结果创建/更新 wiki/concepts/ 下的具体 .md 文件，更没有记录 Evolution Log（强化/修正/分歧）。
4. 处理 personal-writing 的特化流程：
◦ 遗漏：完全没有针对个人写作（不生成 Summary、使用 ## My Position 节、不参与 confidence 的 source_count 计数、Evolution Log 的专属格式）的特殊提取代码。
5. index.md 移动到 Processed 队列及 QUESTIONS 联动：
◦ 遗漏：当前虽然把文件写进了 index.md 的 Sources 列表，但《补充要求.md》明确需要处理 “未处理 (Unprocessed) 到已处理 (Processed)” 的状态变更。并且需要联动检查 wiki/QUESTIONS.md。
￼
二、 依赖工具 qmd 的缺失与命令映射矛盾
6. 查询与反向检验依赖 qmd：
◦ 补充说明中大量提及了 qmd 命令体系（如 qmd add wiki/、qmd status、qmd query "<问题>" --json、qmd multi-get ...等）。
◦ 矛盾：目前我们的环境是一个普通的 Python 脚本环境（如编写了 query.py 占位），并未全局安装或实现 qmd (Karpathy 体系下的终端问答搜索工具) 及其底层向量索引逻辑。
￼
三、 页面模板与文件系统目录的遗漏
7. 特定模板和架构缺失：
◦ 遗漏：虽然《个人知识库系统开发需求规范文档（最终版4.0）.md》及补充文档提到了大量的 wiki/templates/ 模板文件（如 concept-template.md、entity-template.md、personal-writing-template.md、synthesis-template.md 等），但实际的目录 wiki/templates/ 尚未建立并放置这些模板，且脚本生成文件时也没有读取这些模板文件。
8. overview.md 和 QUESTIONS.md 的缺失：
◦ 补充说明要求在系统的根索引文件旁有 overview.md 和 QUESTIONS.md，并在 INGEST/REFLECT 等流程中联动更新。虽然之前的 lint 日志提到了，但文件里的表单格式（如 Health Dashboard、Open Questions 清单等）均未正确初始化。
9. 目录名 ram 与 raw 的拼写矛盾：
◦ 矛盾：补充说明在前面部分写着创建 ram/articles/, ram/pdfs/等，但在实际项目结构及后续描述（“raw/ 层不可变原则”）中都用的是 raw。我们需要统一定义为 raw/。
￼
