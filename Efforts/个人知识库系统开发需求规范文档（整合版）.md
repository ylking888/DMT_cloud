## 1. 项目概述

本项目旨在搭建一个基于 Karpathy LLM Wiki 思路的个人知识库系统，特别强调对扫描版 PDF 的处理能力。通过集成 **LightOnOCR‑2‑1B** 模型，实现高质量的文档识别与公式提取，最终生成符合 Obsidian 规范的 Markdown 文档。

### 1.1 核心目标
- 构建完整的个人知识库系统，支持多种文件类型的自动处理
- 实现扫描版 PDF 的高质量 OCR，尤其是数学公式的识别与 LaTeX 转换
- 生成结构清晰、可直接在 Obsidian 中使用的 Markdown 文档
- 利用 GPU 加速，确保处理效率
- 提供完整的命令行工具链：摄入、检查、索引、反思

### 1.2 技术栈

| 类别         | 技术/库                             | 用途                     |
| ------------ | ----------------------------------- | ------------------------ |
| 核心语言     | Python 3.12                         | 全栈开发                 |
| OCR 引擎     | LightOnOCR‑2‑1B                     | 扫描版 PDF 文字与公式识别 |
| PDF 处理     | pypdfium2                           | PDF 页面渲染为图像       |
| 深度学习框架 | PyTorch, Transformers               | 模型加载与推理           |
| 图像处理     | Pillow, OpenCV                      | 图像预处理               |
| 数学计算     | numpy                               | 数据处理                 |
| Markdown     | 自定义重构引擎 + Python 字符串处理  | 输出 Obsidian 兼容的 MD  |
| 索引工具     | qmd                                 | 文档索引与检索           |
| 配置管理     | PyYAML                              | 解析 YAML frontmatter    |
| GPU 加速     | CUDA 12.0+                          | 模型推理加速             |

### 2.2 核心模块设计

#### 2.2.1 主入口脚本 `ingest.py`

- **职责**：协调各模块，处理命令行参数
- **主要函数**：
  - `main(file_path)`：根据文件类型分发处理
  - `process_pdf(pdf_path)`：PDF 处理（调用 LightOnOCR 包装器）
  - `process_md(md_path)`：Markdown 文件处理
  - `update_system_files()`：更新 index.md、log.md 等
  - `generate_concepts_and_entities()`：从文本中自动提取概念与实体

#### 2.2.2 LightOnOCR 包装器 `lighton_ocr_wrapper.py`

- **职责**：封装 LightOnOCR‑2‑1B 的调用，隔离环境依赖
- **主要函数**：
  - `__init__()`：设置模型路径、Python 环境路径
  - `process_pdf(pdf_path) -> dict`：将 PDF 转为图像，调用模型，返回结构化识别结果
  - `_create_temp_script()`：生成临时 Python 脚本，在原始项目环境中执行
  - `_parse_result(output)`：解析模型输出为统一结构（文本块、公式列表、表格等）

#### 2.2.3 检查脚本 `lint.py`

- **职责**：对知识库进行多项健康检查，生成报告
- **输出**：`wiki/outputs/lint‑YYYY‑MM‑DD.md`

### 3.2 扫描版 PDF 处理流程

1. **PDF 类型检测**：判断是否为扫描版（图片型）还是文本型
2. **页面渲染**：使用 pypdfium2 将每页转为高分辨率图像（≥300 DPI）
3. **版面分析**：识别标题、段落、公式区域、表格区域
4. **OCR 识别**：通过 LightOnOCR‑2‑1B 进行文字识别，同时提取公式 LaTeX 代码
5. **结果结构化**：组装为 JSON 结构，包含文本块、公式列表、表格内容
6. **Markdown 重构**：按阅读顺序生成标题层级、段落、公式（`$...$` / `$$...$$`）、表格
7. **质量验证**：检查公式闭合、特殊字符正确性

## 6. 脚本功能详细需求

### 6.1 `scripts/ingest.py` – 文件摄入

- **输入**：`raw/` 目录下的任意文件路径（PDF / Markdown）
- **输出**：`wiki/sources/` 中对应的 Markdown 文件，更新系统文件
- **核心流程**：
  1. 计算文件 SHA‑256
  2. 若为 PDF，调用 `lighton_ocr_wrapper.process_pdf()`
  3. 获得结构化识别结果（文本、公式、表格）
  4. 生成 Markdown，填写 frontmatter，拷贝至 `wiki/sources/`
  5. 提取概念和实体（可由 AI 辅助），生成对应的概念/实体页面（若不存在）
  6. 更新 `wiki/index.md` 和 `wiki/log.md`
- **扫描版 PDF 特别处理**：
  - 自动检测 PDF 类型
  - 对扫描版使用 LightOnOCR；对文本型直接提取文本（仍可进行公式重识别）
  - 保留原始结构（章节标题、列表等）

### 6.2 `scripts/lint.py` – 健康检查

**10 项检查**：
1. **YAML frontmatter 合法性**：所有 wiki/ 下的 .md 文件必须有合法 YAML，且含 `type` 和 `date`
2. **Broken wikiLinks**：`[[xxx]]` 引用的页面是否存在
3. **Index 一致性**：index.md 中列出的文件是否真实存在
4. **Stubs 页面**：正文少于 100 字的空壳页面
5. **近重复概念**：概念名称（slug）的 Jaccard 相似度 > 0.7 的对
6. **SHA‑256 完整性**：比对 raw 文件哈希与 source 页记录的 `raw_sha256`
7. **Stale 页面**：超过时效阈值的页面（高波动 >90 天，中 =180 天，低 =365 天）
8. **降重审查**：source URL 相似度检测；不同 concept 的 aliases 重叠检测
9. **Wikilink 格式规范**：非英文小写连字符格式、中文词汇、别名错链
10. **公式格式规范**：LaTeX 公式是否正确闭合，是否可渲染

**输出**：`wiki/outputs/lint-YYYY-MM-DD.md`，frontmatter 包含 `graph-excluded: true`

### 6.3 `scripts/lighton_ocr_wrapper.py` – OCR 封装

- 隔离原始项目环境：调用 `LightOnOCR‑2‑1B‑20260329/aiyy.info/` 下的 Python 解释器
- 临时脚本方式执行，避免环境污染
- 返回统一 JSON 结构：
  ```json
  {
    "pages": [
      {
        "page_num": 1,
        "text_blocks": ["段落1", "段落2"],
        "formulas": ["$E=mc^2$", "$$\\int_0^\\infty e^{-x}dx=1$$"],
        "tables": [{"caption": "...", "rows": [...]}]
      }
    ]
  }
  ```

## 7. 公式处理规范

1. **识别**：所有数学公式必须以 LaTeX 格式提取（行内 `$...$`，块 `$$...$$`）
2. **保留**：生成 Markdown 时原样保留 LaTeX 代码，确保 Obsidian 可渲染
3. **分类**：在 source 页面中，核心公式放入「核心公式」章节，其余在正文中展示
4. **校验**：lint.py 检查公式闭合，不完整公式报警
5. **编号**：若原文档有公式编号，可保留为注释 `\tag{1}` 或单独标注

- 系统概述（三层架构、核心原则）
- INGEST 操作规范（含扫描版 PDF 处理流程）
- QUERY 操作规范
- LINT 操作规范
- REFLECT 操作规范
- MERGE 操作规范
- ADD‑QUESTION 操作规范
- Wikilink 使用规范（英文小写连字符，中文别名）
- Wiki 语言规范（中文为主，英文术语可保留）
- Confidence 更新规则（何时提高/降低置信度）
- Source Integrity Rules（原始文件哈希、来源验证）
- 系统文件隔离规则（`graph-excluded: true` 等）
- 文档维护规则（模板、更新日志等）

1. **环境准备**：
   - Python 3.12，CUDA 12.0+

2. **测试**：放入测试 PDF 至 `raw/`，运行 `python scripts/ingest.py raw/test.pdf`

## 11. 风险评估与应对

| 风险               | 等级 | 应对措施                                 |
| ------------------ | ---- | ---------------------------------------- |
| LightOnOCR 模型缺失 | 高   | 部署前校验模型文件完整性                 |

| 处理速度慢         | 中   | 使用 CUDA，优化图像预处理管线            |
